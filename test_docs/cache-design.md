# 缓存系统设计指南

## 概述

缓存是计算机科学中最有效的性能优化手段之一。通过将经常访问的数据存储在高速存储介质中，可以显著降低系统延迟和减少后端负载。本文介绍几种常见的缓存策略及其在知识库系统中的应用。

## LRU 缓存

LRU（Least Recently Used，最近最少使用）是最常用的缓存淘汰策略。其核心思想是：如果一个数据最近被访问过，那么它将来被访问的概率也更高。

### 实现原理

LRU 缓存通常使用双向链表 + 哈希表实现：
- **哈希表**：提供 O(1) 的查找能力
- **双向链表**：维护数据的访问顺序

当数据被访问时，将其移动到链表头部。当缓存满时，淘汰链表尾部的数据。

## 向量近似缓存

在向量检索系统中，缓存不仅限于精确匹配。由于语义相似的查询向量在向量空间中非常接近，我们可以利用这一特性实现近似缓存。

### 近似判断方法

```
1. 计算查询向量的哈希值作为精确匹配 key
2. 对于未匹配的查询，计算与缓存中各向量的余弦相似度
3. 如果相似度 > 阈值（如 0.98），视为缓存命中
```

### 哈希函数

使用 64 位 FNV-1a 哈希将向量压缩为 64 位整数，作为缓存 key。对于降维后的 512 维向量，这种哈希方式碰撞概率极低。

## 查询缓存策略

针对知识库检索场景的缓存策略：

1. **第一层缓存**：精确查询匹配（key 哈希完全一致）
2. **第二层缓存**：语义近似匹配（余弦相似度 > 0.98）
3. **缓存更新**：当检索结果产生后，异步写入缓存
4. **缓存淘汰**：LRU 策略，最大 1000 条

## 混合检索中的缓存优化

在 BM25 + 向量混合检索中，缓存命中后可直接返回融合后的结果，避免重复的数据库查询和向量计算：

```
缓存未命中:
  1. 计算查询向量（1536 维 → 512 维）
  2. BM25 全文检索（PostgreSQL tsvector）
  3. 向量相似度检索（pgvector HNSW）
  4. 融合排序（α * BM25 + (1-α) * 向量相似度）
  5. 写入缓存

缓存命中:
  1. 直接返回缓存结果
  2. 延后更新缓存 LRU 顺序
```

## 性能对比

| 场景 | 无缓存 | 精确缓存 | 近似缓存 |
|------|--------|----------|----------|
| 平均延迟 | 50-200ms | < 1ms | < 5ms |
| 缓存命中率 | 0% | ~20% | ~40% |
| 数据库 QPS 降低 | - | 20% | 40% |

## 总结

合理的缓存设计可以将知识库检索系统的吞吐量提升 2-3 倍，同时将 P99 延迟降低 50% 以上。向量近似缓存特别适合语义搜索场景，能够在保持高召回率的同时大幅提升系统性能。
